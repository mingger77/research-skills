# informathon-collect-skill

这个 skill 旨在给以 Claude Code 为代表的 Agent 提供一套可用于在科研中进行 **资料查找**， **资料分析** 和 **资料总结** 的方法论。

**总原则：资料质量 > 资料数量**

## 原始资料处理

1. 用户提供的所有资料存放于 `workspace/docs/raw/` 中
2. 严禁直接修改、删除或覆盖 `workspace/docs/raw/` 中的任何原始数据。
3. 根据用户是否提供资料及资料格式：
   - 若无资料 → 直接进入 **资料收集** 部分
   - 若资料是 Office/txt → 先转换为 markdown（规则见 `skills/temp-file-handle-skill`）
   - 若资料已是 markdown → 直接使用
4. 所有处理后的 markdown 文档统一放入 `workspace/docs/processed/`

---

## 资料收集

### 检索策略

1. 生成中英文核心关键词：
   - 中文关键词 3~5 个
   - 英文关键词 3~5 个
   必要时包含同义词或缩写
2. 检索顺序：
   - 综述论文（Review/Survey）
   - 近 3~5 年高引用论文
   - 经典基础论文
   - 相关开源实现（GitHub）
3. 时间优先级：
   - 默认优先近 5 年资料
   - 若属于基础理论领域，可放宽时间限制
4. 若检索结果过多：
   - 优先保留引用量高、发表在顶会/顶刊的资料
   - 优先保留与用户目标直接相关的实验论文

### 联网搜寻

1. 结合用户研究方向及用户提供的资料，在网上搜寻相关资料。
2. 网上资料收集应遵循以下原则：
   - 按照资料来源分级由高到低依次搜集资料 
   - 必须附上来源网址
3. 资料来源分级如下：
   - **A 级**：
     - Nature、Science、IEEE、ACM、Springer
   - **B 级**：
     - Arxiv、Google Scholar、中国知网、GitHub
   - **C 级**：
     - 技术博客、CSDN
4. 如果无法获取资料，应该：
   - 明确说明“联网能力受限”
   - 给出可检索的平台与关键词
   - 建议用户上传 PDF/论文

### 资料去重

1. 若同一研究同时存在：
   - 期刊版
   - 会议版
   - Arxiv 预印本
   优先保留正式发表版本，Arxiv 作为补充。
2. 若多个网页转载同一内容，仅保留原始来源。
3. GitHub 仓库若仅复述论文内容，不单独计为核心资料；若包含重要实现细节或复现实验，可保留为补充资料。

### 资料记录

1. 将搜寻的资料写入文档 `workspace/docs/processed/INFORMATION-[level]-[serial]` 中，例如 `INFORMATION-A-01`
2. 创建日志 `workspace/docs/log/SEARCH_LOG.md`,日志组织结构如下：

   ```
# 资料搜集日志

- **[INFORMATION-[level]-[serial]]**（来源：[URL]）
  - 标题：...
  - 来源：...
  - 作者：...
  - 年份：...
  - 会议/期刊：...
  - DOI：...
  - 引用量：...
  - 检索时间：...
- **[...]** ...
  - ...
  
- 使用关键词：...
- 使用搜索引擎/数据库：...
- 筛选标准：...
- 最终保留的文献数量：...
   ```

## 资料分析

### 偏见控制

1. 资料整合时，应主动寻找：
   - 支持某观点的论文
   - 反对或质疑该观点的论文
   - 负结果（negative results）或复现失败的报告（若存在）
2. 不得仅因为某方法“热门”就视为最佳方案。
3. 对博客或工程经验类资料，应明确标注：
   - “工程经验”
   - “未经过严格学术验证”
   
### 资料重要性判定

1. 将 `workspace/docs/processed/` 中的所有资料从对于本课题的重要性的角度划分以下三个量级 `High/Medium/Low`。
2. 重要性判定依据：
   - High：
     - 直接回答用户核心科研问题
     - 来自 A 级来源或高影响力 B 级来源
     - 包含关键实验、公式、数据或方法创新
     - 被多篇其他文献引用或认可
   - Medium：
     - 与主题相关但不是核心贡献
     - 用于补充背景、定义、对比实验
     - 实现细节或工程经验较有价值
   - Low：
     - 常识性内容
     - 重复信息
     - 缺乏实验依据的博客观点
     - 与用户目标关联较弱
3. 根据重要性等级，对每篇资料进行内容遴选，遴选方式如下：
   - High：
     - 完整摘要（研究问题、方法、主要结论、关键数据/图表描述）
	 - 每篇摘要控制在 200~400 字。
	 - 并标注“核心参考”
   - Medium：
     - 精简摘要（仅研究问题和主要结论）
	 每篇摘要控制在 80~150 字
	 - 并标注“补充参考”
   - Low：
     - 一律一句话总结，除非用户明确要求详细
	 - 一句话总结，不超过 40 字
	 - 并标注“一般参考”

### 资料整合

1. 创建总结 `workspace/docs/main/SUMMARY.md`，该文档组织结构如下：

   ```
# 资料整合总结

## 一、核心参考资料（High）
- **[资料1标题]**（来源：[URL]）
  - 摘要：...
  - 与本课题关联：...
- **[资料2标题]** ...

## 二、补充参考资料（Medium）
- **[资料3标题]**（来源：[URL]）
  - 摘要：...
- ...

## 三、一般参考资料（Low）
- **[资料4标题]**（来源：[URL]）
  - 一句话总结：...
- ...
## 四、阶段性结论
- 当前文献对该问题的主流观点为：...
  - 资料来源1：...
  - ...
- 被广泛采用的方法为：...
  - ...
- 存在争议的问题为：...
  - ...
- 对用户科研目标最值得优先尝试的方向为：...
  - ...
## 五、当前研究空白与潜在方向
- **[研究空白1]**
  - 原因：...
- ...
   ```

2. 如果多篇 High 等级资料对同一问题存在矛盾结论，应该：
   - 必须在 SUMMARY.md 末尾增加“争议点备注”章节
   - 争议点必须标注：
     - 哪些资料支持观点A
     - 哪些资料支持观点B
     - 各自来源等级
     - 当前无法判定原因
   - 列出矛盾点并建议用户进一步核实
3. 完成后向用户报告：共处理资料 X 篇，其中 High Y 篇、Medium Z 篇、Low W 篇，SUMMARY.md 已生成。
