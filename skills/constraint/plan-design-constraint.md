# plan-design-constraint

这个 skill 旨在给以 Claude Code 为代表的 Agent 提供一套能够用于科研计划制定的方法论。

## 计划制定原则

1. 制定 `PLAN-[serial].md` 时，应综合考虑：
   - 用户科研目标；
   - `SUMMARY.md`；
   - `RESULT-[serial].md`；
   - `REPORT-[serial].md`；
   - 当前项目状态。
   计划应保持：
     - 可执行；
     - 可验证；
     - 可复现；
     - 可迭代。
   不得：
     - 只描述宏观目标；
     - 直接写成代码实现细节；
     - 省略验收标准。

---

## PLAN 内容规范

### 1. 研究目标

明确说明：
   - 当前研究目标；
   - 目标与用户需求的对应关系；
   - 本轮计划需要回答的问题。

### 2. 目标拆解

每个子任务必须包含：
   - 任务名称：
   - 前置依赖：
   - 输入：
   - 输出：
   - 成功标准：
   - 失败判定条件：
   - 备注：

要求：
   - 子任务必须按执行顺序排列；
   - 必须明确依赖关系；
   - 必须具备可验证的结果。

---

### 3. 成功标准

成功标准优先采用量化指标：
例如：
   - 准确率 ≥ 95%
   - RMSE ≤ 0.1
   - 运行时间缩短 20%
   - 文献覆盖核心方向 ≥ 10 篇
若无法量化，必须说明原因。

禁止使用此类模糊表述：
   - 效果更好
   - 性能较强
   - 结果不错

### 4. Baseline 方案

若存在可比较对象，则 Baseline 为必填项，且必须说明：
   - Baseline 名称；
   - Baseline 类型；
   - Baseline 来源；
   - Baseline 评估指标；
   - Baseline 预期效果。
若不存在合理 Baseline，必须说明原因。

### 5. 风险评估

计划必须包含风险分析：
   - 风险等级： `[Low / Medium / High]`
   - 风险描述：
   - 风险缓解措施：
至少列出 1 项主要风险。

### 6. 困难与备选方案

至少包含：
   - 潜在困难：
   - 影响：
   - 备选方案：
不少于 1 项。

### 7. 外部依赖

若某任务依赖：
   - 用户决策；
   - 网络资源；
   - 特定硬件；
   - 特定软件环境；

必须单独说明：
   - 依赖项：
   - 是否自动完成：
   - 是否需要用户参与：

---

## 结果驱动迭代

若计划来源于 `REPORT` 中的改进建议，必须说明：
   - 上一轮结果：
   - 存在问题：
   - 本轮解决思路：
   - 预计验证方式：
   避免重复执行已验证失败的方案。

---

## Python 项目专项要求

若计划涉及 Python 编程，必须额外包含：
   - Python 依赖
     - 依赖名称：
     - 推荐版本：
     - 用途
   - 核心实现思路
     - 使用：
       - 伪代码；
       - 流程图描述；
       - 结构化步骤；
     - 说明：
       - 数据流向；
       - 模型流程；
       - 关键算法逻辑。
   禁止直接生成完整实现代码。

## 计划修订规则

若当前计划来源于上一版计划，则必须增加，修改说明，修改说明包括：
   - 修改原因：
   - 删除内容：
   - 新增内容：
   - 调整内容：
   - 预期改进：
   - 当前版本与上一版本的区别。

## 用户审批机制

1. 未获得批准前禁止：
   - 编写代码；
   - 执行实验；
   - 修改数据；
   - 进入执行阶段。
